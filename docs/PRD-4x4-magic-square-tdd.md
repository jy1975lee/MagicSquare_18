## 1. Executive Summary

본 프로젝트는 4×4 격자에서 빈칸 2개와 누락 숫자 2개를 고정된 입력/출력 계약 아래 처리하고, 두 가지 값-빈칸 배치 순열을 시도하여 마방진(모든 행·열·주대각 2개의 합이 34)을 만족하는지 판정한 뒤, 성공 시 규칙된 `int[6]`을 반환하는 **순수 로직 훈련** 과제이다. 알고리즘 난이도가 아니라 **불변조건 식별·계약 고정·테스트 가능한 승인 기준·Dual-Track(경계 vs 도메인) TDD**를 동시에 연습하는 것이 핵심이다. 구현은 UI·DB·Web에 의존하지 않으며, Boundary는 “외부에서 들어온 행렬 표현의 검증과 결과/오류의 표준화” 수준으로 정의한다.

## 2. Problem Statement

### 2.1 문제 정의(검증 중심)

표면 과제인 “마방진을 만든다”가 아니라, **주어진 형식의 상태가 고정된 불변조건 집합을 만족하는지 판정하고, 만족 시 그 사실을 고정된 출력 스키마로만 표현한다**는 문제로 정의한다. 성공 경로에서는 (1) 입력 형식·도메인 제약, (2) 빈칸·누락 수의 일관성, (3) 채운 뒤의 마방진 성질(합 34), (4) 시도 순서 및 출력 순서 규칙이 **동시에** 성립해야 한다.

### 2.2 입력/출력 계약이 핵심인 이유

- 판정 기준(행/열/대각선 10개 라인, 상수 34, 빈칸 2개, 값 범위, 중복 부재)이 문장으로만 존재하면 구현자·리뷰어·테스트 간 해석이 달라진다.  
- **입력/출력을 스키마와 순서 규칙까지 단일하게 고정**해야 동일 입력에 동일 출력·동일 실패 코드가 나오고, 자동 테스트가 요구사항의 단일한 해석을 강제한다.  
- 본 PRD의 성공은 “한 번 맞춘 답”이 아니라 **계약 위반 시 거부·해 없음 시 명시된 실패·성공 시 6원소 배열 일치**가 반복적으로 재현되는 것으로 측정한다.

## 4. Scope

### 4.1 In-Scope

- 4×4 정수 행렬에 대한 **입력 검증**(경계 레이어; FR-01)  
- **빈칸 2개**의 1-index 좌표 산출, **row-major(행 우선) 스캔에서 먼저 만나는 0을 첫 번째 빈칸**으로 정의(FR-02)  
- 1~16 중 **누락 숫자 2개** 산출(FR-03)  
- 채워진 4×4에 대한 **마방진 판정**(4행+4열+2대각선 합이 각각 34)(FR-04)  
- **시도 1·시도 2** 순열에 따른 채움·판정 및 **성공 시 `int[6]` 반환 규칙**, **둘 다 실패 시 명시된 오류 정책**(FR-05)  
- **Concept → Rule → Use Case → Contract → Test → Component** 추적 가능성(본 문서 §12)

### 4.2 Out-of-Scope

- **UI 화면** 개발(웹/데스크톱 폼 등)  
- **DB 저장/조회**, 파일 영속화, 네트워크 API 배포  
- **N×N 일반화**(본 PRD의 기본 범위는 4×4만; 일반화는 “확장 항목”으로만 언급 가능, 본 릴리스 요구사항 아님)  
- 마방진 **전수 생성**·해 공간 전체 탐색형 “생성 알고리즘” 과제  
- Report 02에 등장할 수 있는 **Data 레이어(저장/로드) 시나리오** — 본 PRD의 승인 범위에서 제외(저장소 의존 테스트·오류 코드 `E_DATA_*`는 본 문서에 요구하지 않음)

### FR-01 — 입력 검증 (Boundary)

**설명**  
호출자가 제공한 4×4 정수 행렬이 본 PRD의 입력 계약을 만족하는지 검사한다. 실패 시 도메인 솔버로 진행하지 않는다.

**입력**  
- `int[][] matrix`: 행 개수 및 각 행 길이가 모두 4인 4×4 구조로 전달됨을 가정하되, **검증기는 행 개수 4, 각 행 길이 4를 반드시 검사**한다.

**처리 규칙(불변조건)**  
- 행렬 크기: 정확히 4행이고 각 행 길이 4(I1).  
- 값 `0`의 개수: 정확히 2(I2).  
- 모든 셀 값: `0` 또는 `1 ≤ v ≤ 16`(I3).  
- `0`을 제외한 값들은 서로 중복 없음(I4).

**출력**  
- 성공: 검증 통과(반환값은 구현 언어 관례에 따름; “통과”만 계약).  
- 실패: §6 BR 및 §8.1에 정의된 **검증 실패 예외(또는 Result의 오류 분기)** 로 처리하며, **고정된 오류 코드**를 사용한다.

**승인 기준(AC)**  
- AC-FR-01-01: 행 개수가 4가 아닌 입력에 대해 검증이 실패하고, 도메인 솔버가 호출되지 않음을 단언할 수 있다.  
- AC-FR-01-02: 어떤 행의 길이가 4가 아닌 입력에 대해 검증이 실패한다.  
- AC-FR-01-03: `0`의 개수가 0, 1, 3, … 인 입력에 대해 검증이 실패한다.  
- AC-FR-01-04: 17 이상 또는 음수가 포함된 입력에 대해 검증이 실패한다.  
- AC-FR-01-05: `0`이 아닌 동일 값이 2회 이상 등장하는 입력에 대해 검증이 실패한다.

**관련 오류/예외 정책**  
- `E_INPUT_SIZE`: 외곽 크기가 4×4가 아님.  
- `E_EMPTY_COUNT`: `0`의 개수가 2가 아님.  
- `E_VALUE_RANGE`: 어떤 셀이든 `v ∉ {0} ∪ [1,16]`.  
- `E_DUPLICATE_NONZERO`: `0` 제외 값 중 중복 존재.  
- 메시지는 **오류 코드당 고정 문구 1종**(Report 02 §2.4와 동일 취지; 본 PRD에서는 구현 시 해당 표를 단일 소스로 복사·동기화).

---

### FR-02 — 빈칸 탐색 (Domain / Boundary 공통 계약)

**설명**  
값이 `0`인 두 셀의 위치를 1-index 좌표로 산출한다. **첫 번째 빈칸**은 row-major 순서에서 먼저 등장하는 `0`이다.

**입력**  
- FR-01을 통과한 `int[4][4]` 동등 구조.

**처리 규칙(불변조건)**  
- `0`은 정확히 2개(I2).  
- 스캔 순서: 행 인덱스 오름차순, 동일 행 내 열 인덱스 오름차순으로 순회하며 첫 번째·두 번째 `0`의 좌표를 `(r_first, c_first)`, `(r_second, c_second)`로 확정한다.  
- 출력에 사용하는 좌표는 **1-index**(행·열 각각 1~4).

**출력**  
- 내부 표현은 구현 선택이나, FR-05 최종 출력에서 사용하는 `(r1,c1)`, `(r2,c2)`는 위 정의와 일치해야 한다.

**승인 기준(AC)**  
- AC-FR-02-01: FR-01 통과 행렬에 대해 탐색 결과는 정확히 두 좌표 쌍이다.  
- AC-FR-02-02: 첫 좌표는 row-major로 첫 `0`과 일치한다.  
- AC-FR-02-03: 두 좌표는 서로 다르고, 각 `r,c ∈ {1,2,3,4}`이다.

**관련 오류/예외 정책**  
- 빈칸 개수 오류는 FR-01에서 처리; FR-01 이후에는 빈칸 2개가 보장된다.

---

### FR-03 — 누락 숫자 탐색 (Domain)

**설명**  
1~16 중 행렬에 나타나지 않은 두 수를 찾아, **수의 크기 관계가 명확한 쌍**으로 도메인에 전달한다.

**입력**  
- FR-01 통과 행렬.

**처리 규칙(불변조건)**  
- 집합 `{1,…,16}`에서 행렬에 등장한 양수(중복 없음 보장됨)의 여집합은 정확히 2원소(I5).  
- 두 수를 `n_low < n_high`로 명명한다.

**출력**  
- `(n_low, n_high)` 또는 이에 동등한 불변 구조.

**승인 기준(AC)**  
- AC-FR-03-01: FR-01 통과 입력에 대해, 산출된 두 수는 1~16에 속하고 서로 다르다.  
- AC-FR-03-02: 행렬의 비0 원소 집합과 합집합이 `{1,…,16}`이 된다.  
- AC-FR-03-03: `n_low < n_high`가 항상 성립한다.

**관련 오류/예외 정책**  
- FR-01을 통과한 입력에서 누락이 2개가 아닌 경우는 **논리적으로 배제**; 만약 검증 누락으로 도달 시 `DomainRuleError`(또는 동등)로 처리하고 회귀 테스트를 추가한다.

---

### FR-04 — 마방진 판정 (Domain)

**설명**  
4×4의 모든 셀이 1~16으로 채워진 상태에서, 4개 행·4개 열·주대각선 2개(총 10개 라인)의 합이 각각 34인지 판정한다.

**입력**  
- 비어 있지 않은 4×4 정수 행렬(평가 시점에 `0` 없음).

**처리 규칙(불변조건)**  
- 마방진 상수: `MAGIC = 34`(n=4).  
- I7: 위 10개 라인 각각의 합이 34이면 `true`, 하나라도 다르면 `false`.

**출력**  
- 불리언(또는 동등한 명시적 판정 결과).

**승인 기준(AC)**  
- AC-FR-04-01: 알려진 완성 4×4 마방진 표본에 대해 판정이 참이다.  
- AC-FR-04-02: 단일 셀만 34를 깨뜨리도록 변경한 표본에 대해 판정이 거짓이다.  
- AC-FR-04-03: 판정은 10개 라인만 사용함을 테스트로 고정한다(다른 패턴 미사용).

**관련 오류/예외 정책**  
- `0` 포함 행렬에 대한 호출은 계약 위반으로 간주; 호출부 테스트에서 금지.

---

### FR-05 — 해 찾기: 두 조합 시도 및 반환 (Domain + 출력 계약)

**설명**  
누락 두 수 `n_low < n_high`와 빈칸 `(r_first,c_first)`, `(r_second,c_second)`를 사용하여 두 순서로 채운 뒤 FR-04로 검사하고, 성공 시 **고정된 `int[6]`** 을 반환한다. 둘 다 실패하면 **고정 오류 코드**로 실패한다.

**입력**  
- FR-01 통과 행렬(솔버는 내부에서 복사본에 채움을 수행해야 함, §7).

**처리 규칙(불변조건)**  
- **시도 1:** `n_low` → 첫 번째 빈칸, `n_high` → 두 번째 빈칸에 배치한 행렬이 마방진이면 성공. 출력 순서:  
  `[r_first, c_first, n_low, r_second, c_second, n_high]` (1-index).  
- **시도 2(시도 1 실패 시에만 수행):** `n_high` → 첫 번째 빈칸, `n_low` → 두 번째 빈칸에 배치한 행렬이 마방진이면 성공. 출력 순서(역순):  
  `[r_second, c_second, n_low, r_first, c_first, n_high]`  
  즉, **배열 앞쪽 트리플은 두 번째 빈칸·그곳에 배치된 작은 수**, **뒤쪽 트리플은 첫 번째 빈칸·그곳에 배치된 큰 수**이다.  
- **시도 1·2 모두 실패:** 오류 코드 `E_DOMAIN_NO_SOLUTION`으로 실패한다(메시지 고정, §6 BR-13).

**출력**  
- 성공: 위 규칙에 따른 `int[6]`.  
- 실패: `E_DOMAIN_NO_SOLUTION`.

**승인 기준(AC)**  
- AC-FR-05-01: 시도 1만 성공하는 픽스처에 대해 출력이 `[r_first,c_first,n_low,r_second,c_second,n_high]`와 바이트/원소 단위 일치한다.  
- AC-FR-05-02: 시도 1 실패·시도 2만 성공하는 픽스처에 대해 출력이 `[r_second,c_second,n_low,r_first,c_first,n_high]`와 일치한다.  
- AC-FR-05-03: 두 시도 모두 실패하는 픽스처에 대해 `E_DOMAIN_NO_SOLUTION`이 보고된다.  
- AC-FR-05-04: 성공 경로 출력의 길이는 6이고 모든 좌표는 1~4, 모든 값은 1~16이다.

**관련 오류/예외 정책**  
- `E_DOMAIN_NO_SOLUTION`: 고정 메시지 **「두 조합 모두 마방진 조건을 만족하지 않습니다。」** (Report 02와 동일 취지; 구현 시 문자열 단일 상수화).

## 6. Business Rules (도메인·경계)

모든 BR은 **항상 참이어야 하는 규칙**으로 기술한다.

| ID | 규칙 문장 |
|----|-----------|
| BR-01 | 입력 외곽 형태는 정확히 4×4이다(4행, 각 행 길이 4). |
| BR-02 | 셀 값이 `0`인 위치는 정확히 2곳이다. |
| BR-03 | 모든 셀 값은 `v = 0` 또는 `1 ≤ v ≤ 16`이다. |
| BR-04 | `v ≠ 0`인 값들은 서로 다른 정수이다. |
| BR-05 | 첫 번째 빈칸은 row-major 순서에서 첫 번째로 만나는 `0`의 위치이다. |
| BR-06 | 두 번째 빈칸은 동일 순서에서 두 번째 `0`의 위치이다. |
| BR-07 | 마방진 판정에 사용하는 상수는 34이다(4×4 고정). |
| BR-08 | 마방진 성립은 4행·4열·주대각선 2개, 총 10개 라인의 각 합이 34임과 동치이다. |
| BR-09 | 최종 출력의 행·열 좌표는 1-index이며 범위는 1~4이다. |
| BR-10 | 성공 출력은 정확히 6개의 정수를 포함한다. |
| BR-11 | 시도 1에서 첫 빈칸에는 `n_low`, 둘째 빈칸에는 `n_high`를 배치한다. |
| BR-12 | 시도 2 성공 시 출력 순서는 `[r_second,c_second,n_low,r_first,c_first,n_high]`이다. |

## 8. Dual-Track TDD Strategy

### 8.1 Track A — Boundary(UI) TDD

**의미:** 본 PRD 범위에서 “UI”는 **사용자 화면이 아니라** 호출 경계면(입력 수집·검증·도메인 호출 조합·응답/오류 표준화)이다.

**Contract-first 테스트 항목(예시 ID)**  
- BT-01: 유효 입력 시 결과 길이 6 및 좌표 1-index.  
- BT-02: `E_INPUT_SIZE` — 4×4 아님.  
- BT-03: `E_EMPTY_COUNT` — `0` 개수 ≠ 2.  
- BT-04: `E_VALUE_RANGE` — 범위 밖 값.  
- BT-05: `E_DUPLICATE_NONZERO` — 비0 중복.  
- BT-06: 도메인이 `E_DOMAIN_NO_SOLUTION`을 반환할 때 경계가 동일 코드/메시지로 전달.  
- BT-07: 검증 실패 시 도메인 솔버가 호출되지 않음(스파이/플래그 허용 범위는 프로젝트에서 “mock 남용 금지” 원칙 하에 최소화).

**실패 정책**  
- 오류 응답은 `code`(위 enum) + **코드당 고정 `message` 1종**.  
- 동일 `code`에 서로 다른 문구가 존재하면 안 된다.

### 8.2 Track B — Domain(Logic) TDD

**메서드(개념) 단위 테스트 목록**  
- DT-01: 입력 검증(I1–I4) 각각에 대한 실패.  
- DT-02: 빈칸 탐색 row-major 순서.  
- DT-03: 누락 쌍 `n_low < n_high`.  
- DT-04: 마방진 판정 참/거짓.  
- DT-05: 시도 1 성공 경로 출력 형태.  
- DT-06: 시도 2 성공 경로 출력 형태(BR-12).  
- DT-07: 두 시도 실패 → `E_DOMAIN_NO_SOLUTION`.  
- DT-08: 입력 불변성(BR-14).

**불변조건 테스트 목록(매핑)**  
- I1–I4 → FR-01 / DT-01  
- I2, row-major → FR-02 / DT-02  
- I5 → FR-03 / DT-03  
- I6–I7 → FR-04 / DT-04  
- I8–I9(출력 형·순서) → FR-05 / DT-05–DT-07  

### 8.3 병렬 진행 규칙

- **금지:** “도메인을 전부 구현한 뒤 경계 테스트를 추가”하는 순차 단일 트랙.  
- **필수:** 동일 기능 주제에 대해 **Boundary RED & Domain RED를 병렬로 작성**한 뒤, **각각 GREEN**, 마지막에 **REFACTOR**에서 중복 제거 및 의존성 방향 점검.  
- 한 사이클에서 다루는 FR 조각은 **하나의 주제**(예: 크기 검증만)로 제한한다(`.cursorrules` TDD red_phase: 한 번에 여러 기능 혼합 금지).

## 10. Architecture Overview (High-Level)

### 10.1 레이어

| 레이어 | 책임 | 본 PRD에서의 구체 |
|--------|------|-------------------|
| **Boundary** | 외부 표현 검증, 도메인 호출 순서, 성공/실패 응답 표준화 | FR-01, 오류 코드 매핑, “검증 실패 시 솔버 미호출”(BR-15) |
| **Control(선택)** | 유스케이스 조립 | “검증 → 솔브” 한 흐름이면 단일 유스케이스 함수로 충분 가능 |
| **Domain(Entity)** | 빈칸·누락·판정·두 순열 시도 | FR-02~FR-05, BR-01~BR-14 |

### 10.2 SRP / OCP

- **SRP:** 검증, 빈칸 탐색, 누락 탐색, 마방진 판정, 순열 시도를 **별 테스트 군**으로 분리 가능한 단위에 배치한다.  
- **OCP:** N×N 확장은 본 PRD 범위 밖; 4와 34는 **명명된 상수**로만 참조하여 변경 시 한 곳만 수정(하드코딩 산재 금지 — `.cursorrules` forbidden 정신 일치).

### 10.3 의존성 방향

- **Domain은 Boundary를 참조하지 않는다.**  
- Boundary는 Domain을 호출할 수 있다.  
- `.cursorrules` ECB 방향(boundary → control → entity)과 충돌 시 **본 PRD는 “도메인이 경계를 모름”을 우선**한다.

## 11. Risks & Ambiguities

| 결정 항목 | 본 PRD에서의 고정 결정 |
|-----------|------------------------|
| 두 시도 모두 실패 | `E_DOMAIN_NO_SOLUTION` + 고정 메시지(§FR-05). |
| “역순 반환” 의미 | BR-12: `[r_second,c_second,n_low,r_first,c_first,n_high]`. |
| 첫 빈칸 정의 | BR-05 row-major 첫 `0`. |
| 입력 변경 | BR-14: 비변경; 테스트로 강제. |
| Data/저장소 오류 코드 | **Out-of-Scope**; 본 PRD의 Boundary 테스트에 포함하지 않음. |

**자주 잘못되는 포인트**  
- 0-index와 1-index 혼동.  
- “역순”을 값만 바꾼 `[r_first,c_first,n_high,r_second,c_second,n_low]`로 오해 — 본 PRD는 **BR-12**에 따름.  
- 대각선을 한 개만 검사.  
- 입력 행렬 제자리 수정으로 인한 호출자 부작용.

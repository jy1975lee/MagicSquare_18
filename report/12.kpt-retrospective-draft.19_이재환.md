# 4×4 Magic Square — KPT 회고

**문서 목적:** Magic Square(4×4) TDD·ECB 실습을 진행하며 정리한 **Keep / Problem / Try** 회고를 `Report/`에 보관한다.  
**작성일:** 2026-04-28  
**근거 산출물:** `01`~`09` 보고서, `docs/PRD.md`, `.cursorrules`, 구현·커버리지 정리 `08`, 리팩터 분석 `09`

---

## K — Keep (유지할 것)

- **Contract-first:** PRD·설계서 기준으로 입·출력·오류 계약을 먼저 고정한 뒤 구현·테스트에 연결한 흐름.
- **Dual-Track TDD:** Boundary(경계 검증)와 Domain(규칙)을 분리해 RED→GREEN으로 진행한 방식.
- **도메인·경계 회귀 확보:** Domain·Boundary 단위 테스트 보강으로 핵심 규칙 검증 신뢰도를 확보한 점.
- **문서화 루프:** 작업 결과를 `Report/` 등에 남겨 요구–테스트–구현 추적성을 유지한 점.

---

## P — Problem (문제·개선 필요)

- **커버리지 불균형:** 패키지 전체 커버리지와 비교해 GUI(`magicsquare/gui/`)는 자동화 테스트 공백이 큼.
- **리팩터링 부채:** `solve_grid` 등 역할 집중·GUI 초기화·결과 포맷 등 SRP·구조 개선 여지 (상세: `09_(Report)Refactoring_analysis_report.md`).
- **Boundary 검증 중복:** 값 범위·중복 검증 등 유사 순회 로직이 반복되어 변경 비용이 커질 수 있음.
- **문서 동기화 부담:** `docs/`, `Report/`, 프롬프트 산출 간 단일 기준(SSoT) 유지에 노력이 필요함.

---

## T — Try (다음에 시도할 것)

| 시도 항목 | 요지 |
|-----------|------|
| GUI 스모크 테스트 | `pytest-qt` 등으로 윈도우·풀기 버특·오류 메시지 경로 최소 2~3건 추가 |
| 리팩터 1차 (테스트 선행) | `solve_grid` 분해·결과 포맷 매핑 위치 정리·GUI 빌더 메서드 분리 등 계획서 순 반영 |
| 커버리지 게이트 | CI 또는 로컬에서 Domain/Boundary 우선 기준 후 단계적으로 전체 확대 |
| 회고 루틴 | 매 스프린트 말 15분 KPT로 Keep/Problem/Try를 고정 기록 |

## 부록 A — 한 페이지 요약 (발표용)

| 구분 | 내용 |
|------|------|
| **Keep** | 계약 우선, Dual-Track TDD, 도메인·경계 테스트, 문서 추적 |
| **Problem** | GUI 테스트 공백, 구조 부채, 검증 중복, 문서 동기화 |
| **Try** | GUI 스모크·리팩터·커버리지 게이트·주기적 KPT |

## 부록 B — Try 항목 SMART 예시

아래는 첨부 회고 자료의 SMART 원칙에 맞춘 **Try** 문장 예시이다. 실제 일정·환경에 맞게 수치·기한만 조정하면 된다.

| 원칙 | 예시 (본 프로젝트 맥락) |
|------|---------------------------|
| **S**pecific | “테스트를 더 쓴다” ❌ → “`magicsquare/gui/window.py`에 대한 스모크 테스트 3개 추가” ✅ |
| **M**easurable | “커버리지 개선” ❌ → “전체 `magicsquare` 라인 커버리지 72% → 80%” ✅ |
| **A**ttainable | “버그 제로” ❌ → “리팩터 후 회귀 테스트 전부 통유지 + GREEN 테스트 삭제 없음” ✅ |
| **R**ealistic | “기술부채 전부 해결” ❌ → “우선순위 1·2번 리팩터만 다음 스프린트에서 완료” ✅ |
| **T**ime-bound | “언젠가” ❌ → “다음 스프린트(예: 2주) 내 GUI 스모크·게이트 초안 반영” ✅ |

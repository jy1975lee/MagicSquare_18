**작성 기준일:** 2026-04-28

---

## 1. 이번 세션 목표

- Track A(UI/Boundary)와 Track B(Logic)에서 각각 1개씩 RED 테스트를 선택해, 최소 단위로 GREEN 상태를 확인한다.
- GUI는 자동화 테스트를 추가하지 않고, 실행 가능 여부만 1회 수동 스모크로 검증한다.

---

## 2. 선택한 RED 테스트

| Track | RED ID | 설명 | 매핑된 pytest |
|------|--------|------|---------------|
| Track B (Logic) | **L-RED-01** | `find_blank_coords()`가 빈칸 좌표 2개를 row-major 순서(1-index)로 반환 | `tests/domain/test_magic_square_domain_red.py::test_lg_red_01_find_blank_coords_row_major_pair` |
| Track A (UI/Boundary) | **U-RED-01** | 입력이 4x4가 아니면 `E_INPUT_SIZE` 예외 발생 | `tests/boundary/test_magic_square_boundary_red.py::test_ui_red_01_non_4x4_matrix_rejects_with_input_size_error` |

### 선택 이유 (2줄)

- FR-01(입력 형태)과 FR-02(빈칸 좌표 규칙)의 최소 핵심 계약을 동시에 고정할 수 있는 가장 작은 조합이다.
- 이후 `E_EMPTY_COUNT`, 누락 수 계산, 도메인 솔버로 확장하기 전에 경계/도메인의 기초 동작을 안정화할 수 있다.

## 3. GREEN 최소 구현 범위 확인

| 구분 | 내용 |
|------|------|
| Track B | `src/magic_square/domain.py`의 `find_blank_coords()` 동작(빈칸 탐색, row-major, 1-index) 확인 |

| GUI | 기능 확장 없이 실행 스모크만 1회 수행 |

**참고:** 이번 세션에서 소스 코드 내용 변경은 없고, 기존 구현이 선택된 RED 요구를 충족하는지 검증했다.

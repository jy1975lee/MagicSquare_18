## 2. 입력·요구사항(요청 요지)

- 프로젝트: **(4×4) TDD 연습용** — 알고리즘 난이도보다 **Invariant·계약·TDD** 훈련이 목적.
- **입력/출력 계약 고정:** `int[][]` 4×4, `0` 빈칸 정확히 2, 값 0 또는 1~16, 비0 중복 금지, 첫 빈칸 = row-major 첫 `0`; 출력 `int[6]` 1-index `[r1,c1,n1,r2,c2,n2]`, 시도 1·2 규칙 및 실패 시 **명시적 오류 정책** 필수.
- 문서 구조: Executive Summary ~ Traceability Matrix(필수)까지 **지정 목차** 준수.
- **Dual-Track** 문서 반영, **Concept → Rule → Use Case → Contract → Test → Component** 추적성 포함.
- 모호어 금지, 모든 요구사항 **테스트/검증 가능** 문장으로 기술.

## 3. 핵심 결정 사항(보고서에 고정한 내용)

| 주제 | PRD 내 고정 |
|------|----------------|
| **이중 실패** | `E_DOMAIN_NO_SOLUTION` + 고정 메시지(Report 02 취지와 정합) |
| **“역순 반환”(시도 2 성공)** | **BR-12:** `[r_second, c_second, n_low, r_first, c_first, n_high]` — 문서에서 단일 해석으로 못 박음 |
| **입력 불변** | **BR-14:** 호출자 행렬 비변경(내부 복사본에서 채움·판정) |
| **Boundary 범위** | UI·DB·Web **Out-of-Scope**; Track A는 **호출 경계(Boundary)** 계약 테스트로만 정의. Data 레이어 `E_DATA_*`는 본 PRD 요구에서 제외 |
| **대표 픽스처 TD-OK-01** | Report 04 Gherkin 표와 동일 행렬; **시도 1 실패·시도 2 성공**으로 산술 검산 후 기대 출력 **`[4, 4, 1, 3, 3, 6]`** 을 PRD §9.3에 명기 |

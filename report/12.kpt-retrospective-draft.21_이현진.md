1\. Keep (계속 유지할 것들)

무엇이 잘 되었는가? 어떤 관행을 계속 유지해야 하는가? 팀에게 도움이 된 것들은?



K-01 · ECB 의존성 방향을 정적 검증 으로 강제 — 사람의 규율을 코드의 규율로 치환

무엇이 좋았나 : tests/architecture/test\_layer\_dependencies.py 가 boundary → application → domain 외 import 를 CI 차단 (DEP-1 \~ DEP-4).

증거 : 12 / 12 Logic Tasks 가 통과하는 동안 의존성 역전 사고 0 건 · 4 개 Meta 게이트 (TASK-009 \~ 012) 모두 ✅ PASS.

유지 이유 : 클린 아키텍처는 문서가 아닌 테스트 일 때 살아 있다. 코드 리뷰어의 주의력에 의존하면 6 개월 후 첫 위반이 들어온다.

K-02 · Frozen Contract — 입출력 스키마와 에러 메시지를 문자 단위로 동결

무엇이 좋았나 : 9 종 ErrorCode 와 ERROR\_MESSAGES 가 src/magicsquare/boundary/error\_codes.py 단일 모듈에 응고. 변경 시 PR 제목 \[BREAKING-CONTRACT] 강제.

증거 : E-SHAPE-001 \~ E-INTERNAL-001 9 종이 tests/boundary/test\_input\_boundary.py 에서 문자 단위로 비교됨.

유지 이유 : 메시지 한 글자만 흔들려도 다운스트림 파서가 깨진다. "동결" 의 가치를 코드로 표현하는 모범 사례.

K-03 · Decision 필터 (DT-04) — 어휘 규율 을 자동 차단으로 강제

무엇이 좋았나 : "허용 / 유지 / 반환 / 차단 / 계산 / 저장" (Logic) · "보인다 / 가능하다 / 활성화 / 포함한다" (UI) 외 동사로 끝나는 To-Do 는 tests/architecture/test\_decision\_filter.py 가 차단.

증거 : TASK-012 도입 시 TODO.md 의 12 건 위반을 즉시 검출 → 11 건 정정 + \[no-decision-filter] 옵트아웃 마커 1 건만 남김.

유지 이유 : 애매한 동사로 끝나는 To-Do 가 가장 큰 리스크. 어휘를 좁히면 정말 측정 가능한 것만 테스트로 변환된다.

K-04 · Dual-Track TDD — UI/Logic 트랙 독립성 을 어휘로 분리

무엇이 좋았나 : 슬라이드 6.1 / 6.2 의 어휘 분리 규칙을 PRD §11.5 로 정식화. UI 테스트가 domain/ · application/ 을 import 하지 않음을 정적으로 강제 (DT-01).

증거 : src/magicsquare/gui/ (PyQt6 Screen 레이어) 가 composition root 를 통해서만 MagicSquareApp.solve 를 받음.

유지 이유 : 어휘를 분리하면 "내가 지금 어느 트랙에서 일하는지" 가 항상 명료해진다. 트랙이 섞이는 PR 은 자연스럽게 두 PR 로 쪼개진다.

K-05 · ADR 기반 의사결정 — 유보 도 결정이며 AC 가 명시 됨

무엇이 좋았나 : MLOps 도입 욕구를 docs/adr/0001-mlops-triple-track.md 로 Proposed 보류. AC-1 \~ AC-4 미충족이 명시되어 무계획 도입을 차단.

증거 : TASK-017 이 🔒 BLOCKED 상태로 README §8.4 에 추적 매트릭스로 박힘.

유지 이유 : "지금은 안 한다" 를 문서화하지 않으면 같은 토론을 매주 반복하게 된다.

K-06 · Cursor AI 활용 — .cursorrules + .cursor/rules/\*.mdc 분할 정책 으로 모델 가이던스 체계화

무엇이 좋았나 : 단일 .cursorrules (13 KB) 외에 주제별 .mdc 5 종으로 규칙을 분리. AI 의 코드 제안이 ECB 위반·금지 패턴(print(), 매직 넘버, bare except)을 자동으로 회피.

증거 : Forbidden 패턴 검출 0 건으로 12 Tasks 완수.

유지 이유 : "AI 가 알아서 잘함" 이 아니라 "AI 에게 무엇을 금지시킬지" 를 명시한 결과.

K-07 · 11 단계 Report 누적 — PR 단위 회고 가 기록으로 남음

무엇이 좋았나 : Report/1.ProblemDefinition 부터 Report/10.Solver\_Facade\_RED\_GREEN\_Cycle 까지 단계별 의도·시도·교훈을 그대로 보존. 각 Report 는 동명의 Prompting/ 프롬프트와 1:1 매칭.

유지 이유 : 사후 회고 (이 문서) 가 가능한 이유는 누적된 Report 와 프롬프트 카탈로그 덕분.

2\. Problem (문제점들)

어떤 장애물이 있었는가? 무엇이 팀을 느리게 했는가? 어떤 프로세스가 비효율적이었는가?



P-01 · UI Track 진입 BLOCKED — ADR-0002 부재 로 TASK-013 \~ 016 모두 정체

사실 : README §8.3 US-013 \~ 016 이 모두 📦 TODO + 🔒 BLOCKED. 선행 조건 "ADR-0002 — UI 프레임워크 1 택" 이 미작성.

임팩트 : Logic Track 12/12 ✅ 인 반면 UI Track 0/4 📦 — Dual-Track 의 Dual 이 한쪽 다리만 서 있음.

근본 원인 : PyQt6 가 부분 도입되었으나 (src/magicsquare/gui/) 정식 ADR 로 승격되지 않아 다음 PR 의 허가증 이 없음.

P-02 · 로컬 게이트만 가동 — .github/workflows/ci.yml 부재

사실 : tests/architecture/ 4 종 게이트 (의존성·커버리지·매핑표 정합·Decision 필터) 가 모두 로컬에서만 검증됨. README §8.6 다음 PR 후보 Refactoring #3 에 명시.

임팩트 : python -m pytest 를 안 돌린 PR 이 머지될 위험. 12 Tasks 가 ✅ 인 현재 상태 가 push 자동 보장되지 않음.

근본 원인 : Logic Track 완수에 자원이 집중되어 CI 자동화가 후순위로 밀림.

P-03 · Cursor AI 응답 검증 미흡 — 제안을 그대로 수용 한 케이스 산발

사실 : 사용자가 본인 KPT 에서 직접 명시한 항목 ("AI 제안 검증 능력 미흡").

임팩트 : 초기 RED 단계에서 AI 가 제안한 테스트 이름이 Decision 필터를 통과하지 못해 TASK-012 도입 시 11 건 정정 발생.

근본 원인 : AI 제안 → 즉시 수용 패턴. 검증 게이트(린터·테스트·Decision 필터) 가 도입되기 전엔 "그럴듯하면 통과" 가 기본값이었음.

P-04 · 프롬프트 엔지니어링 부족 — 모호한 동사 로 끝나는 초기 To-Do 다발

사실 : Decision 필터 도입 시 12 건 위반 검출 — 도입 이전 의 To-Do 들이 측정 불가능한 동사로 끝나고 있었음을 의미.

임팩트 : "더 견고하게 / 잘 / 안정적으로" 같은 동사가 테스트로 변환되지 못한 채 PR 에 들어옴.

근본 원인 : 프롬프트가 목표 를 적되 허용 동사 집합 을 명시하지 않았음.

P-05 · pyc 캐시 / .pytest\_cache 가 git status 오염

사실 : 회고 시작 시점 git status 에 \_\_pycache\_\_/\*.cpython-310.pyc 60+ 건이 untracked 로 표시됨.

임팩트 : git status 노이즈 → 실제 변경 추적이 어렵고 신규 기여자가 혼란.

근본 원인 : .gitignore 가 \_\_pycache\_\_/ · .pytest\_cache/ 를 충분히 커버하지 않거나 일부 디렉터리가 추가 후 무시 처리됨.

P-06 · M5 진입 전 조기 ML 트랙 욕구 가 학습 부하를 분산할 뻔함

사실 : ADR-0001 §2.4 "학습 부하" — 세 트랙(Logic / UI / ML) 동시 학습 시 각 트랙이 1/3씩 흐려질 리스크가 식별됨.

임팩트 : ADR 로 일단 차단되었으나, 차단되지 않았더라면 어느 트랙도 체화되지 않은 상태 에서 PR 이 양산되었을 위험.

근본 원인 : "다 해보고 싶다" 라는 자연스러운 욕구가 방법론의 깊이 와 충돌.

P-07 · CI 부재로 인한 커버리지 임계 회귀 가능성

사실 : tests/architecture/test\_coverage\_thresholds.py 가 Domain ≥ 95% / Boundary ≥ 85% / Data ≥ 80% 를 검증하나 push 시점 자동 차단이 없음.

임팩트 : 머지 직후 갑작스런 임계 미달이 발견되면 hot-fix 의 cycle time 이 길어짐.

3\. Try (시도해볼 것들)

다음에는 무엇을 다르게 해볼까? 어떤 새로운 도구나 방법을 시도해볼까? 문제 해결을 위한 실험들은?

왜 : P-02 · P-07 동시 해소

T-03 → ACT-03 · Pre-commit Hook 도입 — Black / isort / mypy --strict / Decision 필터

T-04 → ACT-04 · 프롬프트 카탈로그 정식 인덱싱

왜 : P-04 직접 해소 + Cursor AI 호출 시 어휘 게이트가 프롬프트 수준에서 작동

T-05 → ACT-05 · .gitignore 정비 + git rm --cached

무엇 : \_\_pycache\_\_/ · .pytest\_cache/ · .coverage\* 누락 항목 추가, git rm --cached -r 로 untracked 청소

T-06 → ACT-06 · Mutation Testing (mutmut / cosmic-ray) 도메인에 한정 도입

무엇 : tests/domain/ 의 불변식 I1 \~ I6 테스트가 진짜 결함을 잡는지 mutation testing 으로 검증

왜 : 라인/분기 커버리지가 95% 여도 오라클 강도 가 약하면 회귀가 슬쩍 통과할 수 있음

T-07 → ACT-07 · PR 페어 리뷰 게이트 강화 — Self-merge 금지

무엇 : 브랜치 보호 규칙으로 최소 1 인 리뷰 요구, README §10 PR 체크리스트를 GitHub PR 템플릿에 자동 삽입

왜 : P-03 (AI 제안 검증) 의 2 차 방어선

T-08 → ACT-08 · Hypothesis 도입 — 도메인 불변식의 속성 기반 테스트

무엇 : M5 / M6 종료 직후 본 가이드 §0.1 의 90 분 포맷으로 회고 반복

왜 : 회고가 1 회성 이벤트 가 되면 액션 아이템이 누적되어 다음 마일스톤이 무거워짐

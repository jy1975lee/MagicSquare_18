# Magic Square KPT 회고 초안 (v1)

## 문서 목적

- `report/` 하위 산출물(01, 02, 05, 06, 10, 11)을 근거로 이번 세션의 KPT 회고 초안을 정리한다.
- 개인 소감보다는 프로세스/품질 개선 관점에서 다음 액션까지 연결한다.

**작성 기준일:** 2026-04-28

---

## K (Keep) — 계속 유지할 점

- 문제 정의를 결과 중심이 아닌 검증 중심으로 고정한 점  
  - `report/01`에서 불변조건/계약 중심으로 문제를 재정의해 이후 설계와 테스트 방향의 일관성을 확보했다.
- 입력/출력/오류 계약을 문서로 먼저 고정한 점  
  - `report/02`, `report/05`에서 입력 조건, 출력 포맷, 오류 코드를 명시해 구현 전 합의를 선행했다.
- Dual-Track(경계/도메인) 분리 후 최소 RED->GREEN 기준선을 확인한 점  
  - `report/10`에서 Track A/B의 최소 테스트를 선택해 빠르게 동작 기준을 검증했다.
- 문서 간 추적성(Traceability)을 유지한 점  
  - `report/02`, `report/06`의 연결 구조(Invariant->Rule->Use Case->Contract->Test)로 변경 영향 분석이 쉬워졌다.
- 저위험 리팩토링과 보호 테스트를 함께 적용한 점  
  - `report/11`에서 소규모 리팩토링 + 보호 테스트 추가를 통해 회귀 리스크를 통제했다.

---

## P (Problem) — 문제점/아쉬운 점

- 문서/요청 간 경로·대상 불일치가 발생한 점  
  - `report/11` 기준 `gui/main_window.py` 미존재로 `gui/app.py` 대체 분석이 필요했다.
- Task 번호 체계가 이원화된 점  
  - `report/06` 기준 세션 초안 ID와 README 확정 ID가 달라 추적/매핑 비용이 생겼다.
- 실행 환경 의존성으로 GUI 스모크 1차 실패가 있었던 점  
  - `report/10`에서 `PYTHONPATH` 설정 후 해결되었지만 초기에 재현 절차가 충분히 고정되지 않았다.
- RED 스켈레톤 실패가 많은 상태에서 상태 해석이 어려웠던 점  
  - `report/11`에서 실패 수가 유지되는 상황이라 의도된 실패와 회귀 의심을 구분하는 부담이 컸다.
- Boundary 책임/이름의 모호성이 남아 있는 점  
  - `report/11` 분석 결과 `solve()`의 책임 경계가 충분히 명확하지 않았다.

## T (Try) — 다음에 시도할 개선 액션

- 사전 파일·경로 검증 체크리스트를 시작 루틴에 추가  
  - 언제: 다음 세션 시작 5분 이내  
  - 담당: 작업자  
  - 완료 기준: 대상 파일 존재/대체 경로를 세션 시작 문서에 기록
- Task ID 단일화 규칙 수립(README 기준 단일 보드화)  
  - 언제: 다음 README/보고서 업데이트 시  
  - 담당: 작업자  
  - 완료 기준: 최종 관리 ID를 README 체계로 통일하고, 매핑표는 1회성 부록으로만 유지
- 실행 환경 표준화 절차 고정  
  - 언제: 다음 테스트 세션 전  
  - 담당: 작업자  
  - 완료 기준: GUI/pytest 실행 절차를 환경 변수 포함 형태로 문서화하고 1회 재현 성공 확인
- RED 실패 분류표 운영(의도 실패/회귀 의심/환경 이슈)  
  - 언제: 각 테스트 실행 직후  
  - 담당: 작업자  
  - 완료 기준: 실패 항목 100% 분류 및 우선순위 지정
- Boundary/Control 책임 재정리 전 보호 테스트 선행  
  - 언제: 다음 리팩토링 사이클  
  - 담당: 작업자  
  - 완료 기준: `solve` 계약 테스트 선행 작성 후 책임 분리 커밋, 기존 계약 테스트 유지 통과

# Magic Square (4x4) Dual-Track UI + Logic TDD / Clean Architecture 설계 보고서

## 문서 목적
- 본 문서는 구현 코드 없이 설계/계약/테스트/통합 계획만 정의한다.
- 목표는 알고리즘 자체가 아니라 레이어 분리, 계약 기반 테스트, 리팩토링 안정성 훈련이다.
- 입력/출력 계약은 고정하며, 모든 규칙은 테스트로 검증 가능해야 한다.

## 고정 계약

### 입력 계약
- 타입: `4x4 int[][]`
- 빈칸: `0`은 정확히 2개
- 값 범위: `0` 또는 `1~16`
- 중복: `0` 제외 중복 금지

### 출력 계약
- 타입: `int[6]`
- 좌표: `1-index`
- 형식: `[r1,c1,n1,r2,c2,n2]`
- 순서 규칙:
  - `(작은 수 -> 첫 빈칸, 큰 수 -> 둘째 빈칸)` 조합이 마방진이면 해당 순서로 반환
  - 아니면 반대 조합으로 반환

# 1) Logic Layer (Domain Layer) 설계

## 1.1 도메인 개념

| 분류 | 이름 | 책임(SRP) |
|---|---|---|
| Entity | `MagicSquareBoard` | 4x4 상태 보관, 좌표 접근, 값 조회 |
| Value Object | `CellPosition` | 1-index 좌표 `(row,col)` 불변성 유지 |
| Value Object | `MissingPair` | 누락 수 2개 `(low, high)`와 `low < high` 보장 |
| Value Object | `SolveResult` | 출력 포맷 `[r1,c1,n1,r2,c2,n2]` 보장 |
| Domain Service | `BoardInvariantValidator` | 입력 계약 검증(크기/빈칸/범위/중복) |
| Domain Service | `MissingNumberFinder` | `1~16` 기준 누락 수 2개 계산 |
| Domain Service | `MagicSquareJudge` | 행/열/대각선 합(10개 라인) 검증 |
| Domain Service | `TwoPermutationSolver` | 두 조합 시도 후 결과 선택 |

## 1.2 도메인 불변조건(Invariants)

| ID | 개념 | 규칙 | 검증식 |
|---|---|---|---|
| I1 | 행렬 크기 | 정확히 4x4 | `rows==4 && eachRowLength==4` |
| I2 | 빈칸 수 | `0`은 정확히 2개 | `count(0)==2` |
| I3 | 값 범위 | 모든 값은 `0 또는 1~16` | `all(v==0 || (1<=v<=16))` |
| I4 | 중복 금지 | `0` 제외 중복 없음 | `distinct(nonZero) == nonZero.length` |
| I5 | 누락 수 | 누락 수는 정확히 2개 | `missing.length==2` |
| I6 | Magic Constant | 4x4 마방진 상수는 34 | `4*(16+1)/2 == 34` |
| I7 | 마방진 성립 | 4행+4열+2대각선 합 모두 34 | `sum(line_i)==34 for i in 1..10` |
| I8 | 결과 형식 | 길이 6, 좌표 1-index | `len==6 && each(r,c in 1..4)` |
| I9 | 순서 규칙 | low/high 우선 시도 후 반대 시도 | `try(low,high) then try(high,low)` |

## 1.3 핵심 유스케이스(도메인 관점)

| Use Case | 기능 | 입력 | 출력 | 실패조건 |

| UC-D1 | 빈칸 찾기 | Board | `CellPosition[2]` | 빈칸 수가 2가 아님 |
| UC-D2 | 누락 수 찾기 | Board | `MissingPair(low,high)` | 범위/중복 위반 또는 누락 수 2개 아님 |
| UC-D3 | 마방진 판정 | FilledBoard | `boolean` | 판정 불가능 상태 |
| UC-D4 | 두 조합 시도 | Board + MissingPair + EmptyPositions | `SolveResult` | 두 조합 모두 실패 |

## 1.4 Domain API(내부 계약)

| 메서드 시그니처(개념) | 입력 | 출력 | 실패조건 |

| `solveTwoBlanks(matrix:int[4][4])` | matrix | `SolveResult` | 검증 실패 또는 두 조합 실패 시 `SolveFailedError` |

# 2) Screen Layer (UI Layer) 설계 (Boundary Layer)

## 2.1 사용자/호출자 관점 시나리오

| 시나리오 | 행위자 | 행동 | 기대결과 |
|---|---|---|---|
| S1 | 호출자 | 4x4 행렬 전달 | Boundary가 입력 계약 검증 |
| S2 | Boundary | 검증 통과 시 Domain 호출 | 성공 시 Output schema 반환 |
| S3 | Boundary | 검증 실패 | Domain 호출 없이 Error schema 반환 |
| S4 | Boundary | Domain 실패 수신 | 표준 오류 코드/메시지로 매핑 |

## 2.2 UI 계약(외부 계약)

| 구분 | 스키마 | 규칙 |
|---|---|---|
| Input schema | `matrix:number[][]` | 4x4, 빈칸 정확히 2개, 값 범위 0/1~16, non-zero unique |
| Output schema | `result:number[6]` | `[r1,c1,n1,r2,c2,n2]`, 좌표 1-index |
| Error schema | `error:{code,message,details?}` | `code`는 고정 enum, `message`는 고정 문구 |

## 2.3 UI 레벨 테스트(Contract-first, RED 우선)

### 전제
- Domain은 Mock으로 대체한다.
- UI 테스트는 Boundary 계약만 검증한다.

| 테스트 ID | 유형 | 테스트 내용 | 보호 계약 |
|---|---|---|---|
| UT-RED-01 | 형식 | 정상 입력 시 결과 길이 6 | Output schema 고정 |
| UT-RED-02 | 입력 오류 | 4x4 아닌 입력 -> `E_INPUT_SIZE` | Input shape |
| UT-RED-03 | 입력 오류 | 빈칸 수 오류 -> `E_EMPTY_COUNT` | empty count |
| UT-RED-04 | 입력 오류 | 값 범위 오류 -> `E_VALUE_RANGE` | value range |
| UT-RED-05 | 입력 오류 | 중복 오류 -> `E_DUPLICATE_NONZERO` | uniqueness |
| UT-RED-06 | 도메인 오류 | Domain 실패 -> `E_DOMAIN_NO_SOLUTION` | error mapping |
| UT-RED-07 | 형식 | 성공 응답 좌표 1-index 검증 | index rule |

## 2.4 UX/출력 규칙

### 에러 메시지 표준 규칙
- `message`는 고정 문구를 사용한다.
- 동일 오류 코드에는 동일 문구만 허용한다.
- 문구는 원인 1개만 설명한다.
- 문구는 숫자/조건을 명시한다.

| 오류 코드 | 고정 메시지 | 발생 조건 |
|---|---|---|
| E_INPUT_SIZE | 입력 행렬 크기는 4x4이어야 합니다. | shape != 4x4 |
| E_EMPTY_COUNT | 빈칸(0)의 개수는 정확히 2개여야 합니다. | count(0) != 2 |
| E_VALUE_RANGE | 각 값은 0 또는 1~16 범위여야 합니다. | 범위 위반 |
| E_DUPLICATE_NONZERO | 0을 제외한 숫자는 중복될 수 없습니다. | non-zero 중복 |
| E_DOMAIN_NO_SOLUTION | 두 조합 모두 마방진 조건을 만족하지 않습니다. | 두 조합 실패 |
| E_DATA_LOAD_FAILED | 저장소에서 입력을 로드할 수 없습니다. | load 실패 |
| E_DATA_SAVE_FAILED | 저장소에 결과를 저장할 수 없습니다. | save 실패 |

## 3.2 인터페이스 계약

## 3.3 구현 옵션 비교(메모리/파일)

| 옵션 | 구현 | 장점 | 단점 |
|---|---|---|---|
| A | InMemory | 빠른 피드백, 테스트 단순 | 프로세스 종료 시 소실 |
| B | File(JSON/CSV) | 세션 간 유지, 재현성 높음 | I/O 예외, 형식 관리 필요 |

### 추천안
- 추천: **옵션 A (InMemory)** 를 기본 저장소로 채택
- 이유:
  - TDD 학습에서 RED-GREEN-REFACTOR 반복 속도가 가장 중요
  - 외부 I/O 변수를 제거해 실패 원인 분리를 단순화
  - 계약 테스트에 집중 가능
- 보완:
  - 옵션 B는 어댑터 호환성 검증용으로 분리 테스트에서 사용

# 4) Integration & Verification (통합 및 검증)

## 4.1 통합 경로 정의

| 항목 | 규칙 | 검증 방법 |
|---|---|---|
| 의존성 방향 | `UI -> Application(선택) -> Domain <- DataAdapter` | Domain이 UI/Data 타입 미참조 |

| 실패 복구 | Data save 실패 처리 정책 분리 | 경고 반환 또는 오류 반환 정책 테스트 |

## 4. README·브랜치 전략(세션에서 전달한 요지)

- **문서만:** `main` 직접 커밋 또는 `docs/readme-…` 등 짧은 브랜치 + PR.
- **문서+코드:** `feature/…` 한 브랜치에 묶거나, 리뷰 편의를 위해 PR 단위로 분리.
- **과제·팀 규칙:** PR 필수 여부에 맞춰 위 중 선택.

## 5. 후속 권장

- **Dual-Track TDD 구현**은 `feature/dual-track-tdd`에서 이어가고, 안정 시 `develop` 또는 `main`으로 머지하는 흐름을 저장소 규칙에 맞게 정하면 된다.
- **TC-CMM 명세**를 실제 테스트 코드와 맞추려면 언어(Java/Python)·API명·예외 타입을 구현에 맞게 갱신한다.
- 본 세션 이후 추가 커밋이 있으면 **§3의 SHA·브랜치 표**는 해당 시점 기준으로 다시 점검한다.

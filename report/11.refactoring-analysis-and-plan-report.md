## 4. ECB 관점 분석

### 4.1 파일별 현재 역할

| 파일 | 현재 역할 판단 | 메모 |
|------|----------------|------|
| `src/magic_square/domain.py` | Control 성격 우세 | `find_blank_coords()`는 데이터 보관보다 연산 로직 |
| `src/magic_square/boundary.py` | Boundary 부분 충족 | 입력 형태 검증은 적합하나 `solve()`가 실제 제어 위임 없이 미구현 예외 발생 |
| `src/magic_square/gui/app.py` | Boundary(입력) 호출 UI | UI 외 책임(파싱/정규화) 일부 포함 |

### 4.2 책임 재배치 권고

- `boundary.solve()`의 “미구현/해결 불가 판단” 성격은 Control 계층으로 이동 검토
- `gui/app.py`의 입력 파싱·정규화(`"" -> "0"`, `int` 변환)는 Boundary 변환 함수로 이동 검토
- `domain.py`는 데이터 표현과 연산 로직의 모듈 분리(명확한 Entity/Control 경계) 검토

## 6. 리팩토링 대상 목록(우선순위)

| 순번 | 대상 파일 | 문제 | 적용 기법 | 우선순위 |
|---:|---|---|---|---|
| 1 | `src/magic_square/boundary.py` | `solve()`가 실제 제어 흐름 없이 예외만 발생 | Extract Function + 책임 재배치(Boundary는 입출력/검증, Control로 위임) | 높음 |
| 2 | `src/magic_square/gui/app.py` | `_on_solve_clicked()`에 다중 책임 집중 | Extract Method(입력·호출·표시·오류 처리 분리) | 높음 |
| 3 | `src/magic_square/gui/app.py` | `_read_board()` 내 파싱/정규화 규칙이 UI와 결합 | Move Function(입력 변환을 Boundary로 이동) | 중간 |
| 4 | `src/magic_square/domain.py` | Entity/Control 경계 모호 | 모듈 분리(데이터 표현 vs 규칙 실행) | 중간 |
| 5 | `src/magic_square/boundary.py` | 문서상의 크기 표현 하드코딩 | 상수 기반 문서/메시지 일관화 | 낮음 |

## 7. 테스트 선행 필요 항목

- `src/magic_square/boundary.py`의 `solve()`
  - 성공 반환 계약, 실패 시 오류 코드/메시지 계약 먼저 고정 필요
- `src/magic_square/gui/app.py`의 `_read_board()`
  - 빈 문자열 처리, 숫자 파싱 실패, 4x4 변환 결과 검증
- `src/magic_square/gui/app.py`의 `_on_solve_clicked()`
  - 성공 시 결과 라벨 갱신, 실패 시 경고창 메시지 검증
- `src/magic_square/domain.py`의 `find_blank_coords()`
  - 1-index 좌표, row-major 순서, 빈칸 없음/다중 빈칸 케이스 고정

### 8.1 회귀 테스트 실행 명령어

- 전체: `python -m pytest -q`

- 동일 입력 보드에 대해 리팩토링 전/후 `validate()`의 성공/실패 및 오류 코드·메시지 동일성 확인
- GUI 동일 입력 시 결과 라벨 텍스트와 오류 팝업 문구 동일성 확인
- 기존 테스트 + 신규 계약 테스트 전부 통과 확인(특히 예외 경로)

- 코드 수정은 수행하지 않았고, 본 보고서는 이후 구현 단계의 기준선(baseline)으로 사용 가능하다.
